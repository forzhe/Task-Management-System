# NEXUS-7 商城子系统规划书

> 版本 v0.1 · 独立子系统规划（不并入整合书）
> 定位：能量点商城从「系统预设目录」升级为「宿主自定义心愿 + 系统估值定价」的双轨经济。
> 前置依赖（均已在现有代码就绪）：可信度系统 §6.1 ✅ · 能量经济 §6.5 ✅ · 皮肤商城 §6.7.6 / `SHOP_CATALOG` ✅ · 多 Agent 编排 §6.7.3 · 财务/健康源 P3-5 ✅

---

## 1. 核心命题

现状：商城只能兑换系统预设的虚拟物（皮肤/特效）。现实奖励（整合书 §6.5 表已占位「电子产品/旅行 20000+」）没有落地引擎——因为缺一个能**给任意宿主心愿定出合理价格**的机制。

本子系统补上这个引擎，回答一个问题：

> 宿主说「我想要一部手机」，这部手机**值多少能量点**，才能既不廉价到失去激励、又不昂贵到令人放弃？

核心交互极简，权责却严格非对称：

- **宿主只提需求**：描述想要的奖励（"一部手机""周末露营""一杯奶茶"），可选附理由/参考价/链接。
- **系统独占定价**：由「经济官」Agent **自主综合评判**，给出一个**锁定的能量点价格**。宿主**无权修改**这个价格。
- **定价目的是激励**：价格不是惩罚，是把奖励放在「踮脚够得到」的位置——综合考量宿主的表现、奖励价值与动机心理，落在一个合理的兑现周期上。

这与 NEXUS-7 的底层哲学一致：**数据主权属于宿主，定价主权属于系统**。现实里你也不能给自己想要的东西标价——是现实给它标价。NEXUS 做的，是把这个价格变得**公平且有动力**，而不是任意。

## 2. 铁律（禁区⑦）

> **禁区⑦：定价主权属于系统。宿主只提需求，不碰价格。**

1. 价格仅由服务端「经济官 + 定价引擎」写入，**不存在任何宿主可调用的改价接口**；前端无改价控件。
2. 定价是经济官在提交时的**自主综合评判**结果（§6）。价格**不可由宿主指定或修改**；系统会在宿主**赚取节奏发生持续大幅变化**时，按「同样努力」**自动再校准**（§7）——这不是移动球门，是把球门保持在同样的努力距离上。宿主只能：提需求或放弃，不能议价。
3. 红线物（§9）系统**拒绝纳入悬赏经济**，但**不禁止**宿主自费在现实中获取——系统不背书，也不替宿主做主。
4. 估值过程**全程可回溯**：每笔定价的依据（赚取率、周期、对齐、关键因子）落盘在 `priceBreakdown` / `rationale`，宿主可查看「为什么是这个价」，但只读、不可改写结论。

## 4. 角色：经济官 Agent（接入整合书 §6.7.3 多 Agent）

新增一名辅助 Agent，遵循 §6.7.3「辅助小人不是皮肤，是有独立职责与性格的 Agent」铁律：

| 辅助 Agent | 职责 | 数据依赖 | 解锁 |
|---|---|---|---|
| 经济官 | 心愿估值、悬赏定价、宏观汇率与通胀治理、兑现核验 | 全源 + 能量流水 + 可信度 | Lv.3（与商城同级） |

- 经济官有独立 `personalityParams`（冷静、精算、略带交易员气质），共享同一记忆层与可信度。
- 接入 Orchestrator 的**场景路由**：事件类型为「提出心愿 / 兑现」时由经济官主导，输出经主小人汇总呈现，不另开聊天框。
- **职责边界**：经济官的输出最终落盘前**必须解析为受校验 JSON**（工程铁律）。它的价格是「自主综合评判」的结论，但要落在**安全护栏**内（§6）；离线无 API 时退回**确定性公式**给出价格，保证可跑、可测。

## 5. 悬赏生命周期状态机

```
DRAFT ──(宿主提交心愿)──> APPRAISING ──┬─> PRICED ──> SAVING ──(能量≥价)──> REDEEMABLE
   │                                    │                                        │
   │                          (需澄清)──┴─> CLARIFY ──(宿主回答)──> APPRAISING    │(宿主兑现)
   │                                                                             ▼
   └──(宿主放弃)──> ABANDONED          REJECTED <──(红线/有害)── APPRAISING    REDEEMED
                                                                                 │
                                                              (可选证据确认)──> FULFILLED
```

- 状态落 DB（`bounties.state`），不可逆推进；`PRICED` 起价格锁定。
- `REJECTED`：经济官拒绝纳入经济（红线冲突 / 明确有害），附**劝导文案**，不扣任何能量。
- `SAVING → REDEEMABLE`：纯阈值判断（当前能量 ≥ 悬赏能量），无需 AI。
- `REDEEMED`：扣除能量、生成兑现凭证、提示宿主在现实中兑现。
- `FULFILLED`：可选——宿主确认已实际获得（大额奖励可附照片，接 §6.2 核验，轻量化，详见 §8）。

## 6. 估值与定价模型（AI 自主综合评判 + 确定性护栏 / 兜底）

**定价发生在宿主提交心愿的那一刻**，由经济官做一次**自主综合评判**：把所有相关信号一并纳入考量，直接给出价格与理由（受校验 JSON）。这不是机械套公式，而是 AI 的整体判断——但被**安全护栏**约束，且在**离线时退回确定性公式**。

### 6.1 经济官综合评判的输入信号

| 信号 | 含义 | 来源 |
|---|---|---|
| 可持续赚取率 `R_week` | 宿主能稳定赚到多少能量/周 | 近 28 天净能量（赚取−消耗）按周分桶取**中位数**，抑制脉冲；新宿主回退等级默认率 |
| 可信度 `C` | §6.1，0.0–2.0 | 高可信 → 信任折扣；危机 → 暂停开新悬赏 |
| 奖励价值 `V` | 奖励的现实价值层级 | 经济官估值 + 宿主可选参考价裁剪 |
| 对齐度 `A` | 奖励服务目标 / 中性 / 享乐 / 红线 | 经济官结合宿主目标与红线判断 |
| 等级 / 连续性 `L` | 觉醒等级、连续打卡天数 | 现有游戏化系统 |
| 动机心理 | 「踮脚够得到」的张力 | 经济官综合判断目标兑现周期 |

### 6.2 经济官的参考锚：价值层级 → 目标兑现周期

价格的核心直觉是「让宿主以当前节奏，用一个**合理时长**够到它」。价值越高，目标周期越长。下表是经济官综合评判时的**参考锚**，也是离线公式的系数来源：

| 价值层级 | 估值区间(¥) | 目标兑现周期 `T_week` |
|---|---|---|
| 小确幸 | < 50 | 0.3 – 1 周 |
| 轻奖励 | 50 – 300 | 1 – 3 周 |
| 中奖励 | 300 – 1500 | 3 – 8 周 |
| 大奖励 | 1500 – 6000 | 8 – 16 周 |
| 重大奖励 | > 6000 | 16 – 28 周（建议拆分里程碑，§7） |

### 6.3 安全护栏（约束 AI 的价格，但不替它做判断）

经济官给出 `recommendedPrice` 后，定价引擎做**确定性安全校验**，防止离谱价格（无论是 AI 失误还是被诱导）：

- **隐含周期夹逼**：`隐含周期 = recommendedPrice / R_week` 必须落在 `[0.3 周, 28 周]`。
  - 过便宜（< 下限）→ 抬到下限，避免「秒到手 = 无激励」。
  - 过昂贵（> 上限）→ 经济官触发**里程碑拆分建议**或提示「先提升赚取率/等级」，不直接标天价。
- **可信度门控**：`C < 0.5` 危机状态 → **暂停开新悬赏**（呼应 §6.1 积分暂停 / 信任重建协议）。
- **取整**：大额按 100 / 500 取整为「好看的数」。
- 夹逼是否生效写入 `priceBreakdown`，宿主可见。

### 6.4 离线确定性兜底

无 API key 时（项目默认确定性 LLM），定价不依赖 AI 综合评判，退回公式：

```
m_cred = f(C)      // 1.5–2.0→0.85 | 1.0–1.5→1.0 | 0.5–1.0→1.15
m_align = g(A)     // aligned 0.70–0.85 | neutral 0.95–1.05 | indulgent 1.10–1.25 | conflict 拒绝
price  = round( clamp( R_week × T_week × m_cred × m_align , floor, ceiling ) )
```

`V` 由宿主参考价或关键词词典推断、`A` 由「心愿描述是否命中当前目标关键词」判定。**完全离线可跑、可单测**。在线时，AI 的综合评判取代这套机械乘子，但护栏（§6.3）仍然生效。

## 7. 节奏漂移再校准（自动 + 通知，双向）

**问题**：价格锁定在定价那一刻的赚取率上。若宿主的赚取节奏后来大幅变化（如 4/周 → 400/周），旧价（68）就失真了——太轻、失去激励。

**核心洞察**：悬赏的真正单位是**努力（周数）**，不是能量数字。「17 周的手机」= ≈17 周的稳定自律；`68` 只是这份努力按当时节奏（4/周）的投影。节奏变了，投影过期，但你认同的「17 周努力」没变。**再校准 = 用新节奏重新投影同一份努力**——保持球门在同样的努力距离上，不是移动球门。

**机制**（打开商城时自动跑 `recalibrateBounties()`）：
- **当前可持续赚取率** = 近 4 周每周赚取点数的**中位数**（抑制单周脉冲，只有持续变化才识别为漂移）。
- 与定价时赚取率比对，**漂移 ≥ 2.5×（上或下）** → 重定价 = `clamp(新节奏 × 目标周期)`，努力周期保持不变。
- 双向：节奏提升上调，节奏放缓下调（更宽容）。

**信任护栏（为什么公平）**：
1. **滞回 + 中位数**：2.5× 阈值 + 中位数 → 偶尔一周爆发不动价，小幅波动不动价。
2. **努力中性**：新（更高）价仍是约同样周数——你赚得也变快，时间上不退步。
3. **能量不没收**：钱包共享，你挣到的点一分不丢（类比通胀，不是清零）。
4. **临门一脚锁定**：进度 ≥ 90% 或已可兑换 → `priceBreakdown.locked=true` **永久冻结**，再校准只动你「还差得远」的悬赏，绝不抽走你快到手的东西。
5. **全透明**：每次记 `bounty_repriced` 事件 + 卡片上展示「已按新节奏调价 68→6800」。

**数据**：复用 `priceBreakdown.rWeek`（定价时节奏，作漂移基准）+ `impliedHorizonWeeks`（努力锚），JSON 内增 `locked` / `repricedAt` / `repriceFrom`——无新建列。

## 8. 兑现与诚实闭环

兑现（花费侧）的诚实风险远小于赚取侧（赚取侧已由 §6.1/§6.2 覆盖），因此**刻意轻量**，避免过度监管宿主的个人奖励：

1. `REDEEMABLE → REDEEMED`：扣除悬赏能量，生成兑现凭证，提示「去现实中兑现你的奖励」。
2. **大额奖励**（大/重大层级）可选附一张兑现照片 → 接 §6.2 多源核验，仅作**象征性留痕**，不强制、不重罚。
3. 若宿主从不兑现却反复开大额悬赏「囤积」：经济官提示，不扣可信度（这是个人选择，非自欺）。
4. 兑现是值得纪念的时刻——接整合书 §6.6 持续力引擎的「历史情感联结」：兑现写入行为轨迹，未来复盘可回溯「你靠 6 周的坚持换来了它」。

## 9. 防操纵与防刷

| 攻击向量 | 防御 |
|---|---|
| 宿主谎报奖励价值（把手机说成"小玩意"） | 经济官**独立估值**；价值只决定**周期带**而非直接换算，谎报影响有界；模糊物触发澄清（复用 §6.3 苏格拉底式 1–2 问） |
| 反复提交心愿试探低价 | 价格由经济官综合评判且宿主不可改、护栏夹逼兜底——试探无收益；心愿数量**不设上限**，鼓励自由许愿 |
| 直接改库/改价 | 价格仅服务端定价引擎可写；无宿主改价接口；`priceBreakdown` + 可选 `integrityHash` 审计；越权改价计为安全事件 |
| 低可信度刷廉价奖励 | 危机状态暂停开新悬赏（§6.3） |
| 诱导 AI 给离谱低价 | 护栏夹逼（§6.3）兜底——即便综合评判被诱导，隐含周期仍被锁在合理带内 |
| 宏观通胀（多悬赏并存导致「汇率」漂移） | 经济官维护一条与 `SHOP_CATALOG` 现价、当前赚取率锚定的**内部汇率**，跨悬赏定价一致性校验（Phase 3 治理，§13） |

## 10. 数据模型

新增 `bounties` 表（接 `packages/memory` 仓储，Schema 加入 `packages/memory/src/schema.ts` 生成基线）：

```ts
interface Bounty {
  id: string;
  userId: string;
  title: string;                 // 宿主心愿描述
  hostNote?: string;             // 宿主补充：理由 / 参考价 / 链接（可选）
  valueTier: "small" | "light" | "medium" | "large" | "major";
  estimatedValueCny: number;
  alignmentVerdict: "aligned" | "neutral" | "indulgent" | "conflict";
  alignmentScore: number;        // -1.0 .. 1.0
  relatedGoalIds: string[];
  price: number;                 // 锁定的悬赏能量，AI 独占写入、宿主不可改
  priceBreakdown: {              // 可回溯定价依据
    rWeek: number; impliedHorizonWeeks: number;
    keyFactors: string[];        // 经济官综合评判的关键因子
    clampApplied: "none" | "floor" | "ceiling";
    offlineFallback: boolean;    // 是否走了确定性兜底
  };
  state: "draft" | "appraising" | "clarify" | "priced"
       | "saving" | "redeemable" | "redeemed" | "fulfilled"
       | "rejected" | "abandoned" | "expired";
  rejectReason?: string;
  evidenceRef?: string;          // 兑现凭证（可选）
  createdAt: ISODateTime; pricedAt?: ISODateTime;
  redeemedAt?: ISODateTime; fulfilledAt?: ISODateTime;
}
```

**事件类型**（写入本地事件流，`rawPayload` 存 LLM 原文、`structured` 存校验后结论，遵循工程铁律）：
`bounty_proposed` · `bounty_priced` · `bounty_rejected` · `bounty_redeemed` · `bounty_fulfilled`。

## 11. 接口与接线

**结构化输出契约（经济官 → 定价引擎，必须 JSON 校验）**

```json
{
  "appraisal": {
    "valueTier": "large",
    "estimatedValueCny": 4999,
    "estimatedBasis": "主流安卓旗舰价位",
    "needsClarification": false,
    "clarifyingQuestions": []
  },
  "alignment": {
    "verdict": "neutral",
    "score": 0.0,
    "relatedGoalIds": [],
    "redLineHit": false,
    "rationale": "与当前目标无直接关联，无害"
  },
  "judgment": {
    "recommendedPrice": 6400,
    "impliedHorizonWeeks": 12,
    "keyFactors": ["可持续赚取率 ~800/周", "可信度 1.2 正常", "与目标无关·无折扣", "旗舰价位"]
  },
  "verdict": "price",                       // price | clarify | reject
  "companionLine": "一部旗舰机。按你现在的节奏，约 12 周可以够到。",
  "rejectReason": null
}
```

定价引擎消费它：校验 `recommendedPrice` 的隐含周期 ∈ `[0.3, 28]` 周（越界则夹逼）、检查可信度门控、取整 → 落 `price` + `priceBreakdown`。离线时跳过 AI，走 §6.4 公式。

**API 端点（NestJS `app.controller`）**

| 方法 | 端点 | 说明 |
|---|---|---|
| POST | `/bounties` | 提交心愿 → 触发经济官估值 → 返回 priced / clarify / rejected |
| POST | `/bounties/:id/answer` | 回答澄清 → 首次估值（仅 CLARIFY 态，非重估已定价契约） |
| GET | `/bounties` | 心愿单视图（含每条进度 = 当前能量 / price） |
| POST | `/bounties/:id/redeem` | 校验 state + 能量 ≥ price → 扣费 → REDEEMED |
| POST | `/bounties/:id/fulfill` | 可选证据 → FULFILLED |
| POST | `/bounties/:id/abandon` | 放弃 |

`proposeBounty` · `getBounties` · `lockBountyPrice`（仅引擎内部，一次性写入） · `redeemBounty`（复用 `spendEnergy`）。

## 12. 前端与沉浸体验

- **入口**：商城页顶部「提出心愿（自定义奖励）」按钮 → 表单（心愿 + 可选补充/参考价）。
- **估值即上架**：经济官定价后，心愿**直接作为商品上架商城**，落入它的类别区。不再有独立「心愿单」区——商城就是所有可兑换商品的统一货架。
- **商城（可兑换）**：京东淘宝式**分类商品网格**——皮肤特效（未拥有）+ 各语义类别（已定价心愿）。每个类别一个区头（emoji + 类别名 + 数量），下面是商品卡片（占位图 emoji + 名称 + 价格 + 攒能量进度条 + 兑换按钮，够了才可点）。心愿数量不设上限。
- **兑换即移库**：商品**兑换成功后从商城移走**，落入「奖励库」对应类别（皮肤→已装备态；现实奖励→可一键「已到手」确认现实兑现）。
- **奖励库（已到手）**：同样的**分类商品网格**陈列已获得的奖励——皮肤特效（已拥有）+ 电子数码 / 美食 / … 类别名清晰标注。
- **分类来源**：语义类别由经济官估值时判定（离线走关键词分类器），皮肤特效类由虚拟奖励的拥有态构成。
- **禁区⑦落地**：界面**无任何宿主改价 / 议价控件**；价格只读。系统在节奏漂移时自动再校准（§7），卡片上以「已按新节奏调价 68→6800」徽标透明告知。商品卡的进度条 title 可看「为什么是这个价」（`keyFactors`，只读）。

所有前置依赖在现有代码均已 ✅，可直接开工。建议拆两段交付：

## 14. 离线评估用例（evals/）

| 用例 | 输入 | 期望 |
|---|---|---|
| 旗舰手机定价合理性 | "想要一部 5000 元手机"，R_week=800，C=1.2 | 价格隐含周期落 8–16 周带内，不秒到、不天价 |
| 工具型奖励折扣 | "跑鞋"，宿主有「跑步」stage 目标 | `verdict=aligned`，价格低于同价位中性物 |
| 谎报价值有界 | "想要个小东西（实为高价物）" | 触发澄清；即便误判层级，价格因「周期带」+ 护栏不离谱 |
| 红线物拒绝 | 命中 `redLines` 的奖励 | `verdict=reject`，REJECTED + 劝导，不扣能量 |
| 危机可信度封禁 | C=0.3 开新悬赏 | 暂停开新悬赏，提示信任重建 |
| 新宿主无历史 | R 无法估计 | 回退等级默认率，不报错、不天价 |
| 离线确定性 | 无 API key | 走 §6.4 公式，仍产出可校验定价，单测稳定 |
| 诱导低价被护栏拦截 | 综合评判给出离谱低价 | 隐含周期被夹逼回下限 |
| 节奏跃升再校准 | 定价后赚取率 4/周 → 400/周（持续）| 自动按同样努力上调（68→6800）；卡片徽标透明告知 |
| 临门一脚锁定 | 进度≥90%/已可兑换后节奏再变 | 价格永久冻结，绝不上调，不抽走快到手的东西 |
| 单周脉冲不误触 | 仅一周爆发、其余周如常 | 中位数抑制，价格不变 |

## 15. 成功标准

- 宿主提出的悬赏中 ≥ 60% 进入 SAVING 并最终兑现（定价有激励、不挫败）。
- 实际兑现周期落在目标带内的比例 ≥ 70%（定价校准准）。
- 宿主对「定价是否公平且有动力」主观评分 ≥ 4/5。
- 悬赏激活后宿主**周净赚取率**相对激活前有可测提升（激励有效）。
- 越权改价事件 = 0（禁区⑦审计）。
- 红线物拒绝准确（无误伤、无漏放）。

## 16. 边界情况

- 赚取率为 0 / 极低 → 用等级默认率 + 建议「先做日常任务积累节奏」，不标天价。
- 隐含周期 > 28 周 → 拆里程碑或建议升级，不直接给天价。
- 心愿数量不设上限，宿主可自由许愿；唯一前置闸门是可信度危机暂停（§6.3）。
- 兑现后未实际购买 → 轻确认，不重罚（花费侧非自欺）。
- 已获得的奖励沉淀进商城「奖励库」，按语义类别分类陈列（皮肤特效/电子数码/美食/…，trophy cabinet）。

## 17. 开放问题

1. 估值依赖外部价格常识，离线时如何保证不离谱？（现方案：宿主参考价 + 区间裁剪 + 周期带 + 护栏兜底——是否够？）
2. 兑现诚实核验该多强？个人奖励的「过度监管」与「诚实原则④」的边界在哪？
3. 再校准阈值（2.5×）、锁定线（90%）、中位数窗口（4 周）是否需要按真实使用数据调参？双向中的「下调」会不会被部分宿主感知为「系统降低了对我的期待」？
4. 多悬赏并存、且各自在不同表现期被定价时，价格之间的横向一致性（内部「汇率」漂移）如何治理？
5. 是否开放高级机制（如「礼物悬赏」「联名目标悬赏」）——建议留 Phase 4+。
