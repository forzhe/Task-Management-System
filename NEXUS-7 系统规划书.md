# NEXUS-7 个性化人生优化系统 · 详细规划书

> 一个完全由 AI 内核驱动、面向个人成长全场景的任务管理系统。
> 它不是"加了 AI 的 Todo App"，而是"以 AI 为大脑、Todo 是其末梢神经"的人生协作者。

---

## 0. 设计哲学（先确立基本判断）

在动手前，先讲清楚我们与传统任务管理系统（Todoist / Notion / 滴答清单 / Things）的本质差异。如果这条边界守不住，最终就会退化成"GPT 套壳清单"。

### 0.1 三条不可妥协的原则

1. **AI 是主语，不是工具**
   传统系统：用户主动录入任务 → 系统被动存储。
   NEXUS-7：AI 主动观察、提问、提议、复盘；用户在对话中校准方向。任务记录是 AI 的副产物，不是用户的负担。

2. **记忆是核心资产**
   一个不记得你的 AI 不可能帮你成长。系统必须沉淀：你的目标演化、决策模式、情绪曲线、能量节律、长期承诺。这些数据**只服务于这一个用户**，不参与全局训练。

3. **闭环胜过功能**
   宁可少做功能，也要让"目标 → 行动 → 反馈 → 校准"这个最小闭环跑得严丝合缝。一个能让用户三个月后回头看到自己变化的系统，比一个有 50 个 Beta 功能的系统更有价值。

### 0.2 反模式（明确不做什么）

| 反模式 | 为什么不做 |
|---|---|
| 把每件事都游戏化、积分化 | 长期侵蚀内在动机（自我决定理论），让成长变成绩效 KPI |
| 一上来接十种数据源（手环、IoT、Keep…） | 接入工程量大且每个都需要维护，会拖垮迭代节奏 |
| 用炫酷虚拟形象作为核心卖点 | 形式 > 内容，分散打磨"懂用户"这件最重要的事 |
| 用一个超大 Prompt 解决一切 | 不可调试、不可演进、不可控成本 |
| 强制 SOP / 模板化的任务结构 | 个人成长是高度个性化的，模板会异化用户 |

## 1. 用户与场景

### 1.1 目标用户画像

**核心用户：自驱型个人成长者**
- 25–40 岁，有明确的自我提升诉求（职业、健康、关系、财务、创作之一或多个）
- 已经尝试过 2 个以上的工具（Notion / Obsidian / Things…），但都没有坚持下来
- 痛点是"工具有了，但人没变"——系统能记录他们的计划，却不能让他们真的执行
- 愿意为"真的对我的人生产生影响"的服务付费（年付 ¥500–¥2000 区间）

**非目标用户**
- 团队协作场景（已有 Asana / Linear / 飞书）
- 重度结构化笔记爱好者（Notion / Obsidian 更合适）
- 只需要简单提醒的轻度用户（系统对他们过重）

### 1.2 五个核心场景（按发生频率）

| 场景 | 频率 | AI 的角色 |
|---|---|---|
| 日常对话录入 | 每日 5–20 次 | 倾听者 + 结构化记录者：从一句"今天累，没去跑步"中提取情绪、行动、关联目标 |
| 晨间规划 | 每日 1 次 | 教练：基于昨日数据、本周目标、今日日历，提议 3 件最值得做的事 |
| 晚间复盘 | 每日 1 次 | 镜子：呈现今天的能量分布、目标进度、情绪轨迹；问一个直击要害的问题 |
| 周期校准（周/月/季） | 周 1 次 | 战略顾问：识别长期模式、提议调整目标或方法 |
| 临时求助 | 不定期 | 思考伙伴：用户带着具体困扰来（拖延、选择、低能量），AI 用记忆+心理学知识协助 |

## 2. 整体架构（细化版）

在初版 6 层架构基础上，关键调整：

- **新增"记忆与人格档案层"**（位于 AI 中枢和数据层之间，独立成层）
- **AI 中枢从"多引擎"重构为"Agent 编排 + 工具调用"**
- **新增"模型路由器"**控制成本与延迟

## 3. AI 中枢详细设计

这是整个系统的灵魂。下面把"7 个引擎"重构成可工程实现的形态。

### 3.1 Orchestrator（主控 Agent）

**职责**：接收用户消息或定时触发，判断该由哪个职能 Agent 处理，组合多 Agent 输出。
**实现**：一个轻量级 Sonnet 调用，输入 = 用户消息 + 当前上下文摘要 + 工具列表，输出 = 路由决策（JSON）。
**类比**：像 Claude Code 中的主对话 + 子 Agent 调用机制。

### 3.2 六个职能 Agent

| Agent | 触发时机 | 主要工具 | 模型选择 |
|---|---|---|---|
| 对话 Agent | 用户主动发起的非结构化消息 | 读记忆 / 写事件流 / 提取实体 | Haiku |
| 规划 Agent | 晨间触发 或 用户请求"帮我规划" | 读人格档案 / 读目标 / 读日历 / 写任务 | Sonnet |
| 复盘 Agent | 晚间触发 或 周期节点 | 读事件流 / 读情绪 / 读目标进度 / 写洞察 | Opus |
| 洞察 Agent | 后台异步（每周/每月） | 读长期数据 / 模式识别 / 写洞察 | Opus |
| 教练 Agent | 用户求助 / 检测到长期停滞 | 全部读权限 / 苏格拉底式提问 | Sonnet |
| 提醒 Agent | 定时 / 事件触发 | 读上下文 / 选择恰当时机和措辞 | Haiku |

### 3.3 工具集（Agent 可调用的能力）

所有 Agent 通过"工具调用"访问数据，**不直接读写数据库**。这样可以：
- 在工具层统一审计、限流、权限
- 未来更换底层存储不影响 Agent 逻辑
- 方便未来引入 MCP（Model Context Protocol）将工具暴露给其他 AI

**核心工具清单**：

```
读类：
  - get_user_profile()                  → 人格档案
  - query_events(time_range, tags)      → 事件流检索
  - search_memory(query, top_k)         → 语义检索
  - get_active_goals()                  → 当前目标
  - get_calendar(date_range)            → 日程
  - get_energy_pattern(window)          → 能量节律

写类：
  - log_event(event)                    → 写入事件流
  - update_goal(goal_id, delta)         → 目标进度更新
  - create_task(task)                   → 创建任务
  - save_insight(insight)               → 保存洞察
  - update_profile(field, value)        → 更新人格档案（需用户确认）

交互类：
  - send_notification(content, timing)  → 发送通知
  - ask_user(question, urgency)         → 主动提问
```

### 3.4 模型路由策略

成本是个人 AI 系统能否商业化的命门。预估单用户日成本目标：< ¥0.5（重度用户 < ¥2）。

**路由规则**：
- 简单分类、抽取、改写 → Haiku
- 规划、对话主流程 → Sonnet
- 深度复盘、长上下文洞察、月度报告 → Opus
- 用户付费等级影响默认模型（免费版限 Haiku，Pro 版允许 Sonnet/Opus）

**缓存策略**：
- 用户人格档案、长期目标 → Prompt 缓存（命中率应 > 70%）
- 重复触发的日常 Prompt 模板 → 缓存模板部分

## 4. 记忆与人格档案层（核心创新点）

这是 NEXUS-7 区别于其他系统的最关键差异。详细展开。

### 4.1 四类记忆

| 类型 | 存储 | 内容 | 检索方式 |
|---|---|---|---|
| **人格档案** | PostgreSQL JSONB | 价值观 / 长期目标 / 个性偏好 / 红线（如：禁糖、不熬夜后会崩） | 直接查询，几乎每次对话都会注入 |
| **事件流** | ClickHouse | 按时间排序的行动 / 情绪 / 决策 / 对话节选 | 时间窗 + 标签过滤 |
| **关系图谱** | PostgreSQL + 图扩展 | 目标-任务-人物-项目 之间的关联 | 图遍历（"和健身目标相关的所有事件"） |
| **语义记忆** | pgvector | 事件 / 洞察 / 对话片段的向量化版本 | 余弦相似度（"我以前是怎么处理类似的低能量期的"） |

### 4.2 记忆生命周期

```
原始输入（对话/录入/数据同步）
        ↓
   实体抽取（Haiku）
        ↓
   写入事件流（ClickHouse）+ 向量化（pgvector）
        ↓
   按需聚合（每日 → 每周 → 每月）
        ↓
   遗忘策略（90 天后细节降级、保留摘要 + 关键洞察）
```

**关键：必须有遗忘**。不删数据，但对低价值细节做"语义降级"——把"3 月 5 日跑了 3 公里 8 分钟配速"降级为"3 月初保持了规律跑步"。否则上下文会爆炸。

### 4.3 人格档案的演化机制

人格档案不是用户一次填好就不变的，而是在对话和复盘中**渐进演化**：

- AI 在对话中检测到与现有档案矛盾的信号时，提议更新（需用户确认）
- 月度复盘时 Opus 会扫描事件流，识别人格档案中尚未捕获的模式
- 用户可随时打开档案查看 / 修改 / 删除任意条目（**数据主权**）

## 5. 数据模型（核心实体）

只列骨架，详细字段在 spec 阶段补全。

```
User

  ├─ Goals (1..N)

  │    ├─ embeddings
  │    └─ timestamp, tags
  ├─ Insights (AI 生成的洞察)
  │    ├─ scope: daily / weekly / monthly / quarterly
  │    ├─ content
  │    └─ confidence, user_feedback
  └─ Conversations (对话历史)
       └─ messages, tool_calls
```

## 6. 隐私与数据主权

> 个人成长数据 = 一个人最敏感的数据。这一节不是合规栏目，是产品定位本身。

### 6.1 五项承诺（写进产品页和服务条款）

1. **数据所有权归用户**：随时一键导出全部数据（JSON + Markdown），随时一键删除账户及所有派生数据。
2. **不参与训练**：用户数据不用于训练任何全局模型。Agent 个性化通过"上下文注入"实现，不通过微调。
3. **敏感字段端到端加密**：情绪、健康、财务字段使用用户密钥派生加密；服务端持有的也是密文（牺牲一些 AI 处理便捷性以换取信任）。
4. **本地模式（Phase 4 提供）**：核心 Agent 可在用户本地设备运行（用 Haiku 级开源替代），仅在用户主动选择时上云。
5. **零分享默认**：所有"分享"功能默认关闭，用户主动开启时还要二次确认。

### 6.2 法律合规
- 中国大陆：个人信息保护法（PIPL）
- 国际版：GDPR / CCPA
- 关键操作（删除、导出、第三方授权）保留审计日志

## 7. 技术栈选型

| 层 | 选型 | 理由 |
|---|---|---|
| 后端语言 | Python（AI 中枢）+ Go（业务服务） | Python 适合 AI 编排和实验；Go 适合高并发业务接口 |
| AI SDK | Anthropic SDK（Claude） + 可插拔接口 | 主用 Claude（中枢能力强，prompt caching 成熟）；保留切换可能 |
| Agent 框架 | 自研轻量层，参考 Claude Agent SDK 模式 | 重型框架（LangChain/AutoGen）耦合高，自研更可控 |
| 数据库 | PostgreSQL + pgvector | 一份存储搞定结构化 + 向量，运维成本低 |
| 时序事件 | ClickHouse | 事件流的高写入低查询场景最佳选择 |
| 缓存 / 会话 | Redis | 标配 |
| 消息队列 | NATS 或 Kafka | NATS 更轻；用户量 > 10k 后切 Kafka |
| 前端 | React Native（移动） + Tauri（桌面） + Next.js（Web） | 三端一套核心逻辑 |
| 部署 | Kubernetes + Cloudflare（边缘） | 标准云原生 |
| 监控 | OpenTelemetry + Grafana + Sentry | 开源栈 |

## 8. 分阶段实施路线（重排）

把原图的 4 个阶段重新排列，优先级回归"AI 真的懂我"。

### Phase 1 · MVP：闭环可用（0–3 个月）

**目标**：单用户能完成"目标 → 任务 → 行动 → 复盘"的完整一日闭环。

**包含**：
- 对话式输入（文字）
- 手动任务创建 + 系统日历读取
- 3 个 Agent：对话、规划（晨间）、复盘（晚间）
- 基础人格档案（用户首次入职 5 分钟引导问卷）
- 事件流 + 简单语义检索
- 单平台：先 Web，再桌面，移动延后

**不包含**：积分、虚拟形象、多数据源、社交、多用户

**成功标准**：连续 4 周日活率 > 60%，用户能复述出 AI 给他的至少 3 个洞察。

### Phase 2 · 记忆深化与个性化（3–6 个月）

- 增加：洞察 Agent（周/月报告）、教练 Agent（主动求助）
- 增加：关系图谱、语义记忆完整版
- 增加：能量节律识别、情绪追踪
- 增加：移动端 App（对话 + Widget + 通知）
- 增加：第一批数据源插件（笔记 / 健康数据二选一）

**成功标准**：用户在月度复盘后报告"被理解感"高于满意度 4/5。

### Phase 3 · AI 强化（6–12 个月）

- 多模态：语音输入、图片记录、随手拍
- 主动智能：根据上下文主动发起对话，而非只回应
- 更多数据源插件：邮件、IM、IoT（按用户调研优先级排）
- 模型微调实验：基于聚合后的匿名行为数据微调路由模型（不动用户数据）

**成功标准**：被动行为（AI 主动发起的成功对话）占比 > 30%。

### Phase 4 · 生态与进化（12 个月+）

- 虚拟形象 / 沉浸式界面（如果用户调研显示真的需要）
- 本地模式（端侧推理选项）
- 插件市场（用户/第三方贡献的数据源、教练模板）
- MCP 接口暴露（让其他 AI 客户端连接 NEXUS-7 记忆）

## 9. 核心指标体系

避免用 DAU / MAU 这种与"成长"无关的虚荣指标。

| 指标类别 | 指标 | 含义 |
|---|---|---|
| **理解力** | 洞察被用户标记"准"的比率 | AI 对用户的理解是否真实 |
| | 人格档案被用户主动修正的频次 | 越低越好（说明 AI 自己学得对） |
| **执行力** | 用户日均行动条目（不只是创建） | 不是任务数，是真的做了几件事 |
| | 目标完成中位时长 | 长期任务能否被坚持 |
| **关系深度** | 月留存（关键指标） | 不是用户用得勤，是用户离不开 |
| | 用户主动对话占比 | 是否变成"想跟它说话"的关系 |
| **健康度** | 单用户日均 AI 调用成本 | 商业化基础 |
| | P95 响应延迟 | 体验门槛 |

## 10. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| Agent 给出错误建议导致用户决策失误 | 中 | 高 | 所有重大建议标注"AI 建议，请自行判断"；月度复盘明确边界 |
| 用户隐私焦虑导致不愿录入真实数据 | 高 | 高 | 把隐私设计作为产品卖点而非合规栏；提供本地模式 |
| 模型成本失控 | 中 | 高 | 模型路由 + 缓存 + 付费分级 |
| 用户"AI 疲劳"，新鲜感过后弃用 | 高 | 高 | Phase 2 必须证明"被理解感"，否则不进入扩张 |
| Claude API 政策/价格变化 | 中 | 中 | 抽象 LLM 接口层，保留切换能力 |
| 用户把 AI 当心理咨询师，遇到危机内容 | 低 | 极高 | 关键词检测 + 引导专业资源 + 免责声明 |
| 监管对"AI 影响用户决策"提出新合规要求 | 中 | 中 | 关注国内 AI 备案进展，保留可解释性 |

## 11. 开放问题（需进一步决策）

> 这些问题不影响开始，但影响 Phase 2+ 方向。建议在 MVP 进入用户测试后专题讨论。

1. **商业模式**：订阅制？买断制？终身会员？哪种最匹配"长期陪伴"定位？
2. **多用户共享**：夫妻 / 亲子是否需要共享部分上下文？还是严格单人？
3. **AI 的"人格设定"**：是中性专业的教练，还是有温度的伙伴？是否允许用户定制？
4. **失败容忍**：用户连续 N 天不打开，AI 是否主动联系？怎么联系不会显得侵犯？
5. **跨设备一致性**：用户在不同设备上的体验是完全一致的，还是终端差异化（桌面深度 / 手机轻度）？
