## 1. 系统灵魂（最高判据，覆盖一切设计决策）

### 1.1 核心命题

> 系统要持续判断：宿主今天的行为，是在接近理想人生，还是远离理想人生。
> 所有功能都围绕这个问题展开。

这不是 Todo App 加 AI。任务是 AI 的副产物，不是用户的负担。

### 1.2 五条不可妥协原则

**① AI 是主语，不是工具**
AI 主动观察、提问、提议、复盘。用户在对话中校准方向。
违反信号：功能菜单比对话窗口更重要；用户需要手动触发所有 AI 动作。

**② 记忆是核心资产**
一个不记得你的 AI 不可能帮你成长。
必须沉淀：目标演化轨迹、决策模式、情绪曲线、能量节律、长期承诺。
违反信号：重新打开 App 后 AI 不记得昨天发生了什么。

**③ 闭环胜过功能**
"目标 → 行动 → 反馈 → 校准"闭环必须严丝合缝。
一个让用户三个月后看到自己变化的系统，胜过 50 个 Beta 功能。

**④ 诚实胜过讨好**
AI 用于识别"伪完成"、"冲动目标"、"自欺欺人"。
温柔的镜子反映不了现实。必须是冷静的镜子。

**⑤ 沉浸感是工程问题**
外星科技感、虚拟小人、权限解锁、仪式化交互——
这些是经过设计的心理学装置，目标是建立"宿主-系统"深度绑定关系。

### 1.3 六大禁区（明确不做）

1. 粗糙打卡返现式游戏化 → 侵蚀内在动机
2. AI 只鼓励不挑战 → 退化为 ChatGPT 套壳
3. 一个超大 Prompt 解决一切 → 不可调试、不可演进
4. 强制 SOP / 模板化 → 异化用户
5. 数据接入只展示不决策 → 炫技而已
6. 虚拟小人做成皮肤 → 性格必须由 AI 自主调整

## 6. 核心机制详细设计（v0.2 缺口补全）

### 6.1 系统可信度评分算法

**数据类型**：`credibility_score: float`，范围 0.0–2.0，初始值 1.0

**分值变化规则**：

| 事件 | 变化 | 说明 |
|---|---|---|
| 被 AI 标记为"伪完成"后承认 | +0.5 | 诚实奖励 |
| 完成高难度任务（difficulty ≥ 4）并提供证据 | +0.2 | 每次上限 |
| 高质量复盘（AI 评分 ≥ 8/10）| +0.1 | 每日上限 |
| 连续 7 天正常复盘 | +0.3 | 周奖励 |
| 被 AI 标记为"伪完成"后坚持 | -1.0 | 严重惩罚 |
| 逃避复盘（24h 无任何记录）| -0.2 | 每次 |
| 主动修改已完成任务的证据 | -0.5 | 可疑行为 |
| 连续 3 天全部任务显示完成但无任何挣扎记录 | -0.3 | 模式检测 |

**可信度对系统行为的影响**：

| 可信度区间 | 系统响应 |
|---|---|
| 1.5–2.0 | 精英状态。自动信任完成声明；积分 1.5× 加成；小人更信任 |
| 1.0–1.5 | 正常状态。任务完成需要证据备注 |
| 0.5–1.0 | 警戒状态。所有任务完成需要 AI 二次确认；小人更严格 |
| 0.0–0.5 | 危机状态。积分暂停；系统提醒；强制进入"信任重建协议" |

**可信度显示**：不直接显示数字，通过小人状态和系统"权限等级"隐性反馈。

### 6.2 自欺识别：Windows 多源校验决策树

**触发条件**：任何任务状态标记为"completed"时，或复盘中主观声称某行为时

```
输入：宿主声明（例："今天专注学习了 2 小时 Claude Code"）

Step 1. 取同时段客观数据（时间窗口：任务 scheduledAt → completedAt 或当日）
  - ActivityWatch.窗口数据：VSCode/终端前台时长
  - ActivityWatch.AFK 数据：键鼠活跃时长  
  - 浏览器历史（Phase 1+）：同时段 URL 分类
  - Obsidian 文件变更（如涉及学习）：新增/修改笔记时间戳

Step 2. 计算"客观支持度"
  客观支持度 = (客观专注时长 / 声明时长) * 数据来源数

  - ≥ 0.8 → TRUSTED，直接完成
  - 0.4–0.8 → UNCERTAIN，标记"待确认"，Request 证据补充
  - < 0.4 且数据来源 ≥ 2 → SUSPICIOUS，触发复盘 Agent 对比

Step 3. SUSPICIOUS 路径
  复盘 Agent 生成诚实问题："系统注意到一些不一致的信号。
  当天 VSCode 前台时间为 26 分钟，键鼠活跃 31 分钟。
  能描述一下 2 小时里具体做了什么吗？"

Step 4. 用户响应
  - 提供合理解释（如：使用其他设备、手写笔记）
    → Decision Agent 评估解释可信度
    → 可信 → TRUSTED；不可信 → 降为 UNCERTAIN

  - 承认高估 → credibility_score +0.5 + 标记真实状态
  - 坚持完成 → credibility_score -1.0 + 标记"伪完成（已存疑）"

Step 5. 数据未接入时的降级策略
  - ActivityWatch 未运行：UNCERTAIN，仅要求证据备注
  - 浏览器历史未接入：降低置信权重，不触发 SUSPICIOUS
```

**小人触发文案示例（SUSPICIOUS 时）**：
> "数据链路显示今日专注记录与声明存在偏差。系统不会强制修正——但诚实是最高属性。"

### 6.3 目标真实性检测：苏格拉底式对话流

**触发条件**：用户新建目标 或 教练 Agent 检测到目标停滞 > 14 天

**对话轮次设计（教练 Agent 执行）**：

```
Round 1 — 背景追溯
  问："是什么事情让你今天想到这个目标？"
  目的：区分长期真实意图 vs 一时情绪冲动

Round 2 — 代价确认
  问："如果实现这个目标，你的日常生活会具体改变什么？
       有什么事情你愿意为此放弃？"
  目的：评估宿主是否真正接受目标代价

Round 3 — 历史校对
  系统注入：过去 90 天中，同类型目标的完成率、停滞原因
  问："系统注意到你过去 [X 次] 设置过类似目标，
       [Y 次] 在 [Z 天] 后停止推进。这次有什么不同？"
  目的：对抗乐观偏见

Round 4 — 明确化（仅当通过前三轮时）
  问："把目标改写成可测量的形式——
       在 [时间段] 内，具体做到 [可验证的行为指标]"
  系统辅助建议："例如：30 天内每天学习 2 小时，学习日记可证明"
  目的：将模糊目标具象化

Round 5 — 冲动概率评估（后台，不展示给用户）
  AI 综合评分输出：
  {
    "authenticity_score": 0-100,  // 真实性分数
    "impulse_probability": 0-1,   // 冲动可能性
    "recommended_action": "proceed" | "defer_3days" | "reframe",
    "breakdown": { "timing": ..., "commitment": ..., "history": ... }
  }
  
  impulse_probability > 0.7 → 系统建议："NEXUS 建议将此目标暂存 3 天后重新确认"
  宿主可接受建议或强制创建（强制创建记入档案）
```

### 6.4 Evolution Agent 安全设计

**唯一权限**：可提议修改其他 Agent 的 Prompt 模板（修改 `prompt_registry`），
            不可修改数据结构、不可修改 Safety Agent、不可修改自身 Prompt。

**运行时机**：每周一次，后台异步，不影响实时响应。

**修改前必须满足的三个条件**：

1. **离线评估集验证**：被修改的 Prompt 必须在 `evals/` 中有对应评估集，
   修改后的版本在评估集上得分不低于当前版本
2. **渐进升级**：新版本先在 10% 的触发场景中 A/B 测试 7 天，
   测试期间老版本保持运行
3. **用户可见且可回滚**：所有修改记录到 `system_evolution_logs`，
   用户可在"进化日志"中查看修改内容和原因，可一键回滚

**绝对禁区（硬编码，不可被 Evolution Agent 修改）**：
- Safety Agent 的任何 Prompt 或逻辑
- 可信度评分算法（§6.1）
- 数据写入规则
- 用户数据删除权限

**Evolution Agent 评估指标**（每次运行时观察）：
- 用户对 AI 回复的 👍/👎 比率（近 7 天）
- 任务完成率变化趋势
- 档案被修正频率（频率高 = AI 理解不准）
- 复盘 Agent 洞察被用户确认率

### 6.5 积分经济模型

**能量点来源（量化）**：

| 来源 | 基础积分 | 备注 |
|---|---|---|
| 完成普通任务（difficulty 1-2）| 10–25 | - |
| 完成中等任务（difficulty 3）| 30–60 | - |
| 完成困难任务（difficulty 4）| 80–150 | 需 AI 验证证据 |
| 完成极难任务（difficulty 5）| 200–500 | 需 AI 验证 + 宿主确认 |
| 完成周目标 | 300–800 | 按目标重要度 |
| 完成月目标 | 1000–3000 | - |
| 高质量日终复盘 | +20 | AI 评分 ≥ 8/10 |
| 连续打卡加成 | × (1 + 连续天数×0.02) | 上限 1.5× |
| 主动抵抗诱惑（记录并承认）| +30 | 特殊奖励 |
| 诚实承认"伪完成" | +50 | 诚实奖励 |

**四层防刷机制**：

1. **主观确认**：完成时必须写证据备注（> 10 字）
2. **数据核查**：ActivityWatch 数据对照（Step 2 可信度）
3. **成果验证**：困难任务需要截图/链接/Obsidian 笔记引用
4. **AI 判断**：复盘 Agent 异步对全天任务做整体一致性检验

**商城兑换规则**（Phase 1 实施）：

| 类型 | 示例 | 积分范围 | 防刷要求 |
|---|---|---|---|
| 现实奖励（小）| 奶茶、零食 | 500–2000 | 自定义验证（拍照）|
| 现实奖励（中）| 电影、游戏时间 | 3000–8000 | 需周目标达成 |
| 现实奖励（大）| 电子产品、旅行 | 20000+ | 需月目标达成 + AI 确认 |
| 虚拟奖励 | 小人皮肤、特效 | 1000–5000 | 可信度 ≥ 1.0 |
| 功能奖励 | 高级分析、路径模拟 | 10000+ | 等级 ≥ 20 |

### 6.6 持续力引擎（v1.0 新增：从"记录"到"坚持"的转化机制）

**问题诊断**：截至 Phase 1 末，系统擅长记录"昨天发生了什么"，但对"明天还会不会继续"
的影响力很弱。记录是必要条件，不是充分条件。长期坚持需要四个系统此前缺失的机制：
连续感、早期反馈、历史情感联结、关键时刻的主动在场。

**设计判据**：每个机制都必须回答——"在宿主想放弃的那个普通的周三下午，这个机制起什么作用？"

#### 6.6.1 习惯链系统（Streak Chain）

**定义**：某行为类别在连续自然日内至少发生一次，即维持一条"链"。

**跟踪类别**（Phase 2 首批）：

| 类别 | 判定事件 |
|---|---|
| 晨间规划 | trigger=morning_planning 的事件 |
| 日终校准 | category=daily_review 的事件 |
| 任意任务完成 | toStatus=completed 的任务事件 |
| 特定目标推进 | relatedGoalIds 含该目标的任务完成事件 |

**数据模型**：`user_streaks` 表

```
category        text      -- 链类别
goal_id         text?     -- 仅"特定目标推进"使用
current_streak  integer   -- 当前连续天数
longest_streak  integer   -- 历史最长（永不清零，是荣誉资产）
last_active_date text     -- 最后活跃日（YYYY-MM-DD）
broken_at       text[]    -- 断链日期数组（供洞察 Agent 分析断链模式）
```

**结算规则**：
- 自然日 24:00 结算；当日无该类事件 → 断链，`current_streak` 归零，断链日期入 `broken_at`
- 不设"修复券"之类的宽限机制——断了就是断了（诚实原则 ①④），但 `longest_streak` 永久保留
- 断链不是惩罚事件，是数据事件：断链后 24h 内触发洞察微分析（见 6.6.2）

**展示要求**：
- 主界面常驻显示活跃链（"晨间规划 · 连续 7 天"）
- 链中断后显示"上次连续 12 天，断于 6-10"——断裂感本身是信息，不隐藏
- 小人台词必须感知链状态（见 6.6.3）

#### 6.6.2 主动洞察触发器（Proactive Insight）

**问题**：InsightAgent 目前是被动按钮 + 每周一定时。但维持坚持最需要反馈的时刻
是第 3 天、第 7 天——数据还不够"周报"级别的时候。

**触发矩阵**：

| 触发条件 | 响应级别 | 模型 | 形式 |
|---|---|---|---|
| streak 达到 3 / 7 / 14 / 21 / 30 / 60 / 100 | 微洞察 | haiku | 一句话 + 小人庆祝状态 |
| streak ≥ 7 后断链 | 断链分析 | sonnet | 3 句以内：断链前的模式变化 + 一个具体重启建议 |
| 连续 3 天任务完成率 < 30% | 低谷检测 | sonnet | 小人关怀对话 + 提议 coach session |
| 每周一 09:00（保留现有）| 周洞察 | opus | 完整洞察报告 |

**微洞察的内容要求**：必须引用具体历史，不说空话。
> 反例："坚持得很好，继续加油！"
> 正例："连续 7 天晨间规划——你上一次做到这一点是 4 月初，那次在第 9 天断了。这周末是关键。"

**成本控制**：微洞察用 haiku 档（约 ¥0.002/次），日均增量成本 < ¥0.01。

#### 6.6.3 小人历史记忆（Companion Memory）

**问题**：小人台词由 AI 生成但没有"历史感"——它不记得宿主上周差点放弃、上个月的低谷。
没有历史感就没有情感联结；没有情感联结，"有个存在在看着你"就不成立。

**数据模型**：`companion_memories` 表

```
type         text   -- low_point | near_quit | recovery | peak | promise
summary      text   -- 一句话事实描述（"6-3 连续 4 天零完成后，6-7 主动重启晨间规划"）
occurred_at  text
ref_event_ids text[] -- 关联事件，可溯源
emotional_weight real -- 0-1，注入优先级
```

**写入时机**：日终校准后，由 ReviewAgent 顺带判断"今天是否构成关键时刻"
（不增加独立 LLM 调用，复用复盘那次调用的输出新增一个字段 `keyMoment?`）。

**注入规则**：CompanionAgent 每次调用注入：最近 3 条 + emotional_weight 最高 2 条（< 150 token）。

**台词要求**：高情绪时刻（庆祝/低谷/断链）必须引用历史记忆。
> "上周三你也说不想动。那天你最后还是做了 25 分钟。今天呢？"

#### 6.6.4 关键时刻主动介入（Critical Moment Intervention）

**问题**：长期坚持的敌人不是大失败，是某个什么都不想做的普通下午——而那一刻没有东西来找你。

**信号检测**（SchedulerService 每小时轻量检查，纯 SQL 判断，不调 LLM）：

| 信号 | 检测条件 |
|---|---|
| streak_at_risk | 19:00 后当日任务完成数 = 0 且任意 streak ≥ 3 |
| stalled_task | 任务 in_progress 停留 > 2h 无状态变化 |
| review_at_risk | 22:00 未做日终校准且校准链 ≥ 3 |
| silent_day | 全天无任何事件写入（含周末，但文案区分）|

**介入预算（不骚扰原则）**：
- 每日最多 2 次主动通知；同一信号每日只触发 1 次
- 介入升级阶梯：单次信号 → haiku 一句话提醒（经 ReminderAgent）；
  同类信号一周内第 3 次 → 提议 coach session（"要不要聊聊是什么在挡着你？"）
- 介入永远是邀请不是命令；宿主连续 3 次忽略同类介入 → 该信号静默 7 天并记入档案
  （"宿主在 X 场景下不希望被打扰"——这本身是档案数据）

**与可信度系统的关系**：介入频率不因可信度低而增加。低可信度通过 §6.1 的验证强度
处理，不通过骚扰处理——惩罚和陪伴是两个独立通道，混用会同时毁掉两者。

### 6.7 全域感知引擎（Phase 3 设计：从"屏幕前的你"到"完整的你"）

**问题诊断**：Phase 1-2 让系统懂了"屏幕前的你"——它的判断锚点是 ActivityWatch 屏幕数据、
浏览器历史和你的主观陈述。但理想人生不止发生在屏幕前：身体、睡眠、财务、线下行动、人际
同样定义你是谁。系统的判断只能和它的输入一样好。当感知有盲区，"接近还是远离理想人生"
这个核心判断就有系统性偏差，而且这些盲区恰恰是最容易自欺的地方（"我今天很健康""我没乱花钱"）。

**Phase 3 核心命题**：把系统的**感官**从单一屏幕扩展到多源生活信号，把 AI 的**视角**从单一
主小人扩展到多个专精 Agent，从而对"接近还是远离理想人生"给出更立体、更难自欺的判断。

**设计判据**：每个新数据源都必须回答——"它喂给哪一个决策？" 只展示不决策的源不接（禁区⑤）。

#### 6.7.1 数据源抽象层（DataSource 统一契约）

所有新源实现同一接口，归一化后写入事件流（复用现有 event schema 的 `source`/`category`/`confidence`/`tags`）：

```
interface DataSource {
  kind: "health" | "finance" | "calendar" | "mail";
  mode: "oauth" | "csv_import" | "local_rest";
  fetch(window): NormalizedSignal[];   // 拉取并归一化
}
```

| 源 | 接入方式 | 喂给哪个决策（必填）|
|---|---|---|
| 健康/运动（小米手环/Keep/Strava CSV、Google Fit）| CSV 导入 / OAuth | 体力属性增长、净成长值、健康类目标真实性校验 |
| 财务（账单 CSV）| CSV 导入 | 红线检测（冲动消费）、目标代价评估（§6.3 Round 2）|
| 日历（Microsoft Graph / .ics）| OAuth / 文件 | 晨间规划上下文、时间冲突、复盘客观锚点 |
| 邮件摘要（IMAP / Graph）| OAuth | 承诺/deadline 提取、目标停滞检测 |

优先级：健康/运动 CSV 最易验证（无 OAuth、有现成导出），作为第一个真实源打通链路。

#### 6.7.2 跨源自欺识别 v2（N 源校验决策树）

扩展 §6.2 的三源决策树到 N 源，引入**源权重**与**置信聚合**：

```
输入：宿主声明（例："今天锻炼了 1 小时，状态很好"）
Step 1. 收集同时段所有可用源信号（屏幕/浏览器/健康/财务/日历）
Step 2. 每源给出"支持 / 中立 / 矛盾"判定 + 源可靠性权重
        （客观传感器 > 第三方 API > 手动导入 > 主观陈述）
Step 3. 加权聚合：
        - 多源一致支持 → 置信叠加，TRUSTED
        - 多源一致矛盾 → SUSPICIOUS（例："Keep 无记录、手环步数 1200、心率平稳"）
        - 源之间冲突 → 按权重裁决，必要时进入黑箱裁决（§6.7.4）
Step 4. SUSPICIOUS → 复盘 Agent 引用具体多源数据点名，不软化
```

#### 6.7.3 多 Agent 协同（辅助 Agent，不是皮肤）

**铁律（禁区⑥）**：辅助小人不是皮肤，是有独立职责与性格的 Agent。

| 辅助 Agent | 职责 | 数据依赖 | 解锁 |
|---|---|---|---|
| 健康管家 | 身体维度：运动/睡眠/久坐提醒，体力属性守护 | 健康源 | Lv.20 |
| 学习教练 | 认知成长：学习类事件深度、知识复利 | 屏幕/浏览器/Obsidian | Lv.25 |
| 复盘镜子 | 强化版独立人格的诚实之镜（现 review 升格）| 全源 | Lv.5（已有）|

**编排升级**：Orchestrator 从"消息路由"升级为"场景路由"——按事件类型 + 当前上下文决定
哪个 Agent 主导。多 Agent 输出**由主小人汇总呈现**，避免界面割裂成多个聊天框。每个辅助
Agent 有独立 `personalityParams`，但共享同一记忆层与可信度。

#### 6.7.4 黑箱裁决机制

当 N 源客观数据与宿主主观严重冲突（加权置信差 > 阈值）且宿主坚持原陈述时：

1. 系统**不强制修正**——尊重宿主数据主权
2. 进入"黑箱裁决"：记录这条分歧（claim vs 多源证据），可信度按 §6.1 处理
3. **持续追踪**：后续相关事件会被关联到这条分歧，验证它最终被证实还是证伪
4. 全程透明可回溯，宿主可在"分歧档案"查看/申诉

黑箱裁决是诚实原则（④）的制度化：系统记得你坚持过什么，以及现实后来站在了哪一边。

#### 6.7.5 语义记忆 v2（真向量召回）

多源接入后事件量级跃升，关键词召回不够用。接入 embedding（DeepSeek/OpenAI embedding API；
offline 时退回 Phase 1 的关键词评分版）。`searchMemory` 升级为**向量优先 + 关键词兜底**，
事件写入时异步计算 embedding 存入 `events.embedding`（schema 已预留该列）。

#### 6.7.6 经济模型落地（皮肤商城首批）

§6.5 商城从占位变可用：能量点兑换小人皮肤/特效（虚拟奖励，门槛 可信度 ≥ 1.0）。
这是 Lv.3 商城的第一批真实内容，也让能量点第一次有真实去处。

## 8. 游戏化系统

### 8.1 六维属性（每个维度独立经验条）

| 属性 | 增长来源 | 衰减触发 |
|---|---|---|
| 智力 | 学习/阅读/写作/思考任务 | 14天无此类任务 → 轻微衰减 |
| 体力 | 运动/睡眠合规/饮食记录 | 7天无运动 → 停止增长 |
| 专注力 | 深度工作时长（ActivityWatch 核实）| 分心比率 > 60% 连续 3 天 → 衰减 |
| 意志力 | 克服困难任务/抵抗诱惑记录 | 连续逃避复盘 → 衰减 |
| 创造力 | 写作/设计/构想/Obsidian 笔记 | 无明显触发（创造需保护）|
| 秩序感 | 规划执行/复盘完成/目标追踪 | 逃避复盘/任务长期拖延 → 衰减 |

**衰减策略（避免挫败，重在提醒）**：

```
3 天未触发 → 停止增长（不减）
7 天未触发 → 轻微衰减（-5%）
14 天未触发 → 提醒 + 小人关怀对话 + 衰减（-10%）
30 天未触发 → 目标冻结建议 + 衰减（-20%）
```

### 8.2 等级公式

```
觉醒等级 = f(目标完成度, 属性成长, 连续性, 复盘质量, 长期稳定性)

具体：
  base_xp = Σ(任务奖励积分 × 属性加成)
  multiplier = 1 + (连续打卡天数 × 0.02) + (复盘质量均值 / 100)
  weekly_xp = base_xp × multiplier
  level = floor(sqrt(total_xp / 100))  // 每 100 XP 在 1 级，越高越难
```

| OpenAI（GPT）| `OPENAI_API_KEY` | gpt-4o-mini / gpt-4o / o1 |
| Gemini（Google）| `GEMINI_API_KEY` | gemini-2.0-flash / gemini-2.0-flash / gemini-2.5-pro |
| DeepSeek | `DEEPSEEK_API_KEY` | deepseek-chat / deepseek-chat / deepseek-reasoner |
| Kimi（Moonshot）| `KIMI_API_KEY` | moonshot-v1-8k / moonshot-v1-32k / moonshot-v1-128k |
| 离线（确定性）| 无需 key | 固定回复，用于开发/测试 |

**切换方式**（`.env` 示例）：
```
# 使用 DeepSeek
NEXUS_LLM_PROVIDER=deepseek
DEEPSEEK_API_KEY=sk-xxx

# 自动检测（有哪个 key 用哪个，优先级：anthropic → openai → gemini → deepseek → kimi）
NEXUS_LLM_PROVIDER=auto
```

**覆盖单个 tier 的模型**（不改 provider，只换某一个档位的模型）：
```
NEXUS_LLM_MODEL_OPUS=deepseek-reasoner   # 把 opus 档换成 deepseek-reasoner
NEXUS_LLM_MODEL_HAIKU=gpt-4o-mini        # 把 haiku 档换成 gpt-4o-mini
```

**tier 与 Agent 的对应关系**（由 `ModelRouter` 决定，与 provider 无关）：

| Agent | 档位 | 原因 |
|---|---|---|
| 复盘、洞察、进化 | opus | 深度推理，高质量输出 |
| 规划、对话、Profile、决策 | sonnet | 平衡质量与成本 |
| 小人台词、提醒、分类 | haiku | 响应速度优先 |

**成本控制策略**：
- Phase 0–1：优先用 DeepSeek / Kimi（性价比最高，国内直连无需代理）
- Phase 2+：Anthropic Claude 的 Prompt Cache 可大幅降低重复 context 的成本
- Phase 4：本地 Ollama 承担 haiku/sonnet 档，仅 opus 级任务走 API（目标 < ¥0.3/日）
