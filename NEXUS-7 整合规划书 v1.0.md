## 1. 系统灵魂（最高判据，覆盖一切设计决策）

### 1.1 核心命题

> 系统要持续判断：宿主今天的行为，是在接近理想人生，还是远离理想人生。
> 所有功能都围绕这个问题展开。

这不是 Todo App 加 AI。任务是 AI 的副产物，不是用户的负担。

### 1.2 五条不可妥协原则

**① AI 是主语，不是工具**
AI 主动观察、提问、提议、复盘。用户在对话中校准方向。
违反信号：功能菜单比对话窗口更重要；用户需要手动触发所有 AI 动作。

**② 记忆是核心资产**
一个不记得你的 AI 不可能帮你成长。
必须沉淀：目标演化轨迹、决策模式、情绪曲线、能量节律、长期承诺。
违反信号：重新打开 App 后 AI 不记得昨天发生了什么。

**③ 闭环胜过功能**
"目标 → 行动 → 反馈 → 校准"闭环必须严丝合缝。
一个让用户三个月后看到自己变化的系统，胜过 50 个 Beta 功能。

**④ 诚实胜过讨好**
AI 用于识别"伪完成"、"冲动目标"、"自欺欺人"。
温柔的镜子反映不了现实。必须是冷静的镜子。

**⑤ 沉浸感是工程问题**
外星科技感、虚拟小人、权限解锁、仪式化交互——
这些是经过设计的心理学装置，目标是建立"宿主-系统"深度绑定关系。

### 1.3 六大禁区（明确不做）

1. 粗糙打卡返现式游戏化 → 侵蚀内在动机
2. AI 只鼓励不挑战 → 退化为 ChatGPT 套壳
3. 一个超大 Prompt 解决一切 → 不可调试、不可演进
4. 强制 SOP / 模板化 → 异化用户
5. 数据接入只展示不决策 → 炫技而已
6. 虚拟小人做成皮肤 → 性格必须由 AI 自主调整

1. Profile Agent 每周扫描事件流，识别行为与档案的矛盾

4. 月复盘时 Opus 做深度人格扫描，可能提议新红线/激励偏好
5. 用户随时可查看/修改/删除任意条目（**数据主权**）
6. 所有档案变更可回滚

## 6. 核心机制详细设计（v0.2 缺口补全）

### 6.1 系统可信度评分算法

**数据类型**：`credibility_score: float`，范围 0.0–2.0，初始值 1.0

**分值变化规则**：

| 事件 | 变化 | 说明 |
|---|---|---|
| 被 AI 标记为"伪完成"后承认 | +0.5 | 诚实奖励 |
| 完成高难度任务（difficulty ≥ 4）并提供证据 | +0.2 | 每次上限 |
| 高质量复盘（AI 评分 ≥ 8/10）| +0.1 | 每日上限 |
| 连续 7 天正常复盘 | +0.3 | 周奖励 |
| 被 AI 标记为"伪完成"后坚持 | -1.0 | 严重惩罚 |
| 逃避复盘（24h 无任何记录）| -0.2 | 每次 |
| 主动修改已完成任务的证据 | -0.5 | 可疑行为 |
| 连续 3 天全部任务显示完成但无任何挣扎记录 | -0.3 | 模式检测 |

**可信度对系统行为的影响**：

| 可信度区间 | 系统响应 |
|---|---|
| 1.5–2.0 | 精英状态。自动信任完成声明；积分 1.5× 加成；小人更信任 |
| 1.0–1.5 | 正常状态。任务完成需要证据备注 |
| 0.5–1.0 | 警戒状态。所有任务完成需要 AI 二次确认；小人更严格 |
| 0.0–0.5 | 危机状态。积分暂停；系统提醒；强制进入"信任重建协议" |

**可信度显示**：不直接显示数字，通过小人状态和系统"权限等级"隐性反馈。

### 6.2 自欺识别：Windows 多源校验决策树

**触发条件**：任何任务状态标记为"completed"时，或复盘中主观声称某行为时

```
输入：宿主声明（例："今天专注学习了 2 小时 Claude Code"）

Step 1. 取同时段客观数据（时间窗口：任务 scheduledAt → completedAt 或当日）
  - ActivityWatch.窗口数据：VSCode/终端前台时长
  - ActivityWatch.AFK 数据：键鼠活跃时长  
  - 浏览器历史（Phase 1+）：同时段 URL 分类
  - Obsidian 文件变更（如涉及学习）：新增/修改笔记时间戳

Step 2. 计算"客观支持度"
  客观支持度 = (客观专注时长 / 声明时长) * 数据来源数

  - ≥ 0.8 → TRUSTED，直接完成
  - 0.4–0.8 → UNCERTAIN，标记"待确认"，Request 证据补充
  - < 0.4 且数据来源 ≥ 2 → SUSPICIOUS，触发复盘 Agent 对比

Step 3. SUSPICIOUS 路径
  复盘 Agent 生成诚实问题："系统注意到一些不一致的信号。
  当天 VSCode 前台时间为 26 分钟，键鼠活跃 31 分钟。
  能描述一下 2 小时里具体做了什么吗？"

Step 4. 用户响应
  - 提供合理解释（如：使用其他设备、手写笔记）
    → Decision Agent 评估解释可信度
    → 可信 → TRUSTED；不可信 → 降为 UNCERTAIN

  - 承认高估 → credibility_score +0.5 + 标记真实状态
  - 坚持完成 → credibility_score -1.0 + 标记"伪完成（已存疑）"

Step 5. 数据未接入时的降级策略
  - ActivityWatch 未运行：UNCERTAIN，仅要求证据备注
  - 浏览器历史未接入：降低置信权重，不触发 SUSPICIOUS
```

**小人触发文案示例（SUSPICIOUS 时）**：
> "数据链路显示今日专注记录与声明存在偏差。系统不会强制修正——但诚实是最高属性。"

### 6.3 目标真实性检测：苏格拉底式对话流

**触发条件**：用户新建目标 或 教练 Agent 检测到目标停滞 > 14 天

**对话轮次设计（教练 Agent 执行）**：

```
Round 1 — 背景追溯
  问："是什么事情让你今天想到这个目标？"
  目的：区分长期真实意图 vs 一时情绪冲动

Round 2 — 代价确认
  问："如果实现这个目标，你的日常生活会具体改变什么？
       有什么事情你愿意为此放弃？"
  目的：评估宿主是否真正接受目标代价

Round 3 — 历史校对
  系统注入：过去 90 天中，同类型目标的完成率、停滞原因
  问："系统注意到你过去 [X 次] 设置过类似目标，
       [Y 次] 在 [Z 天] 后停止推进。这次有什么不同？"
  目的：对抗乐观偏见

Round 4 — 明确化（仅当通过前三轮时）
  问："把目标改写成可测量的形式——
       在 [时间段] 内，具体做到 [可验证的行为指标]"
  系统辅助建议："例如：30 天内每天学习 2 小时，学习日记可证明"
  目的：将模糊目标具象化

Round 5 — 冲动概率评估（后台，不展示给用户）
  AI 综合评分输出：
  {
    "authenticity_score": 0-100,  // 真实性分数
    "impulse_probability": 0-1,   // 冲动可能性
    "recommended_action": "proceed" | "defer_3days" | "reframe",
    "breakdown": { "timing": ..., "commitment": ..., "history": ... }
  }
  
  impulse_probability > 0.7 → 系统建议："NEXUS 建议将此目标暂存 3 天后重新确认"
  宿主可接受建议或强制创建（强制创建记入档案）
```

### 6.4 Evolution Agent 安全设计

**唯一权限**：可提议修改其他 Agent 的 Prompt 模板（修改 `prompt_registry`），
            不可修改数据结构、不可修改 Safety Agent、不可修改自身 Prompt。

**运行时机**：每周一次，后台异步，不影响实时响应。

**修改前必须满足的三个条件**：

1. **离线评估集验证**：被修改的 Prompt 必须在 `evals/` 中有对应评估集，
   修改后的版本在评估集上得分不低于当前版本
2. **渐进升级**：新版本先在 10% 的触发场景中 A/B 测试 7 天，
   测试期间老版本保持运行
3. **用户可见且可回滚**：所有修改记录到 `system_evolution_logs`，
   用户可在"进化日志"中查看修改内容和原因，可一键回滚

**绝对禁区（硬编码，不可被 Evolution Agent 修改）**：
- Safety Agent 的任何 Prompt 或逻辑
- 可信度评分算法（§6.1）
- 数据写入规则
- 用户数据删除权限

**Evolution Agent 评估指标**（每次运行时观察）：
- 用户对 AI 回复的 👍/👎 比率（近 7 天）
- 任务完成率变化趋势
- 档案被修正频率（频率高 = AI 理解不准）
- 复盘 Agent 洞察被用户确认率

### 6.5 积分经济模型

**能量点来源（量化）**：

| 来源 | 基础积分 | 备注 |
|---|---|---|
| 完成普通任务（difficulty 1-2）| 10–25 | - |
| 完成中等任务（difficulty 3）| 30–60 | - |
| 完成困难任务（difficulty 4）| 80–150 | 需 AI 验证证据 |
| 完成极难任务（difficulty 5）| 200–500 | 需 AI 验证 + 宿主确认 |
| 完成周目标 | 300–800 | 按目标重要度 |
| 完成月目标 | 1000–3000 | - |
| 高质量日终复盘 | +20 | AI 评分 ≥ 8/10 |
| 连续打卡加成 | × (1 + 连续天数×0.02) | 上限 1.5× |
| 主动抵抗诱惑（记录并承认）| +30 | 特殊奖励 |
| 诚实承认"伪完成" | +50 | 诚实奖励 |

**四层防刷机制**：

1. **主观确认**：完成时必须写证据备注（> 10 字）
2. **数据核查**：ActivityWatch 数据对照（Step 2 可信度）
3. **成果验证**：困难任务需要截图/链接/Obsidian 笔记引用
4. **AI 判断**：复盘 Agent 异步对全天任务做整体一致性检验

**商城兑换规则**（Phase 1 实施）：

| 类型 | 示例 | 积分范围 | 防刷要求 |
|---|---|---|---|
| 现实奖励（小）| 奶茶、零食 | 500–2000 | 自定义验证（拍照）|
| 现实奖励（中）| 电影、游戏时间 | 3000–8000 | 需周目标达成 |
| 现实奖励（大）| 电子产品、旅行 | 20000+ | 需月目标达成 + AI 确认 |
| 虚拟奖励 | 小人皮肤、特效 | 1000–5000 | 可信度 ≥ 1.0 |
| 功能奖励 | 高级分析、路径模拟 | 10000+ | 等级 ≥ 20 |

## 8. 游戏化系统

### 8.1 六维属性（每个维度独立经验条）

| 属性 | 增长来源 | 衰减触发 |
|---|---|---|
| 智力 | 学习/阅读/写作/思考任务 | 14天无此类任务 → 轻微衰减 |
| 体力 | 运动/睡眠合规/饮食记录 | 7天无运动 → 停止增长 |
| 专注力 | 深度工作时长（ActivityWatch 核实）| 分心比率 > 60% 连续 3 天 → 衰减 |
| 意志力 | 克服困难任务/抵抗诱惑记录 | 连续逃避复盘 → 衰减 |
| 创造力 | 写作/设计/构想/Obsidian 笔记 | 无明显触发（创造需保护）|
| 秩序感 | 规划执行/复盘完成/目标追踪 | 逃避复盘/任务长期拖延 → 衰减 |

**衰减策略（避免挫败，重在提醒）**：

```
3 天未触发 → 停止增长（不减）
7 天未触发 → 轻微衰减（-5%）
14 天未触发 → 提醒 + 小人关怀对话 + 衰减（-10%）
30 天未触发 → 目标冻结建议 + 衰减（-20%）
```

### 8.2 等级公式

```
觉醒等级 = f(目标完成度, 属性成长, 连续性, 复盘质量, 长期稳定性)

具体：
  base_xp = Σ(任务奖励积分 × 属性加成)
  multiplier = 1 + (连续打卡天数 × 0.02) + (复盘质量均值 / 100)
  weekly_xp = base_xp × multiplier
  level = floor(sqrt(total_xp / 100))  // 每 100 XP 在 1 级，越高越难
```

### 10.1 AI Provider 多模型支持

系统通过 `LlmClient` 抽象层与具体模型解耦，**切换 provider 只需改 `.env`，代码零修改**。

**支持的 Provider**：

| Provider | 环境变量 | 默认模型（haiku / sonnet / opus）|
|---|---|---|
| Anthropic（Claude）| `ANTHROPIC_API_KEY` | claude-haiku-4-5 / claude-sonnet-4-6 / claude-opus-4-8 |
| OpenAI（GPT）| `OPENAI_API_KEY` | gpt-4o-mini / gpt-4o / o1 |
| Gemini（Google）| `GEMINI_API_KEY` | gemini-2.0-flash / gemini-2.0-flash / gemini-2.5-pro |
| DeepSeek | `DEEPSEEK_API_KEY` | deepseek-chat / deepseek-chat / deepseek-reasoner |
| Kimi（Moonshot）| `KIMI_API_KEY` | moonshot-v1-8k / moonshot-v1-32k / moonshot-v1-128k |
| 离线（确定性）| 无需 key | 固定回复，用于开发/测试 |

**切换方式**（`.env` 示例）：
```
# 使用 DeepSeek
NEXUS_LLM_PROVIDER=deepseek
DEEPSEEK_API_KEY=sk-xxx

# 自动检测（有哪个 key 用哪个，优先级：anthropic → openai → gemini → deepseek → kimi）
NEXUS_LLM_PROVIDER=auto
```

**覆盖单个 tier 的模型**（不改 provider，只换某一个档位的模型）：
```
NEXUS_LLM_MODEL_OPUS=deepseek-reasoner   # 把 opus 档换成 deepseek-reasoner
NEXUS_LLM_MODEL_HAIKU=gpt-4o-mini        # 把 haiku 档换成 gpt-4o-mini
```

**tier 与 Agent 的对应关系**（由 `ModelRouter` 决定，与 provider 无关）：

| Agent | 档位 | 原因 |
|---|---|---|
| 复盘、洞察、进化 | opus | 深度推理，高质量输出 |
| 规划、对话、Profile、决策 | sonnet | 平衡质量与成本 |
| 小人台词、提醒、分类 | haiku | 响应速度优先 |

**成本控制策略**：
- Phase 0–1：优先用 DeepSeek / Kimi（性价比最高，国内直连无需代理）
- Phase 2+：Anthropic Claude 的 Prompt Cache 可大幅降低重复 context 的成本
- Phase 4：本地 Ollama 承担 haiku/sonnet 档，仅 opus 级任务走 API（目标 < ¥0.3/日）
